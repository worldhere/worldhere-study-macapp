# 任务时长自动推算设计

## 背景

药品任务包（Excel）只有 3 列：机型、任务名称、条数。没有时长字段。

一个班次大约做 3000 条。组长无法给出单个任务预估时长，只知道总体速率。

需要能从任务包快速出排班结果，结果"看起来对得上"，不必追求精确。

## 目标

- 导入无时长字段的任务包时，自动推算合理的预估时长
- 不同任务时长有一定区分度（基于名称关键词），不是全部等长
- 自动分配产出的排班结果接近实际整体状态

## 设计

### 1. 时长推算公式

```
预估时长(分钟) = 条数 × (班次有效分钟数 ÷ 班次速率) × 关键词系数
```

| 参数 | 默认值 | 说明 |
|------|--------|------|
| 班次有效分钟数 | 420 | 白班扣除休息后的可用分钟 |
| 班次速率 | 3000 | 口述值，可在设置页调整 |
| 每条耗时 | 0.14 分钟 | 420 / 3000，自动计算 |
| 关键词系数 | 1.0 | 见下表 |

每条耗时 = 420 / 3000 ≈ 0.14 分钟 → 25 条 × 0.14 = **3.5 分钟基础时长**

### 2. 关键词系数表

根据任务名称中的动作词判断复杂度，累计叠加（上限 1.5）：

| 系数 | 动作关键词 | 含义 |
|------|-----------|------|
| 0.8 | 并排、紧贴、移到、推向、移到、移到、放、横放 | 简单位移类 |
| 1.0 | 放入、抽出、打开、合上、倒出、放进、放中间 | 普通操作 |
| 1.2 | 拧紧、摇晃、晃动、翻转、整理、挤出、插入 | 精细/费力操作 |
| 1.5 | 多个复杂关键词叠加 | 复合操作 |

叠加规则：
- 每匹配到一个关键词，乘以对应系数
- 最终系数上限 1.5，下限 0.7
- 如果没匹配到任何关键词，使用 1.0

### 3. 新增函数

在 `utils.py` 新增 `estimate_duration_from_name(name: str, count: int, rate_per_shift: int = 3000) -> int`:

```
计算每条耗时 = 420 / rate_per_shift
基础时长 = count × 每条耗时
扫描名称关键词，累积系数
返回 round(基础时长 × clamp(系数, 0.7, 1.5))
```

### 4. 导入流程增强

`import_utils.py` 的 `execute_import()` 中：
- 如果任务 duration 为空且 name 不为空，调用 `estimate_duration_from_name()` 自动填入
- 计算结果写入 `duration`（字符串，如"3.5"）和 `est_seconds`（整数秒）
- `est_mode` 设为 `"auto"` 标记自动推算

### 5. 设置页新增配置项

在 `config` 表新增一行（settings 面板已有管理界面）：

| category | key | value | description |
|----------|-----|-------|-------------|
| schedule_params | rate_per_shift | 3000 | 每班次采集条数 |

提供配置入口供后续调整。

### 6. 自动分配

现有 `auto_assign.py` 行为不变。无优先级时，所有任务等同对待，算法按"最早可用机器×最早空闲时段"贪心分配。

## 涉及文件

| 文件 | 改动 |
|------|------|
| `utils.py` | 新增 `estimate_duration_from_name()` |
| `import_utils.py` | `execute_import()` 中调用推算函数 |
| `routes/settings.py` | 支持 `rate_per_shift` 配置的读写 |
| `templates/panels/settings.html` | 新增 `rate_per_shift` 输入项 |

## 不变更

- 不修改 Excel 导入的字段映射逻辑
- 不修改自动分配算法
- 不修改数据库表结构（使用现有 tasks.duration / tasks.est_seconds 字段）
