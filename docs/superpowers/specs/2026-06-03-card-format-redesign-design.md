# 飞书推送卡片格式改造 — 设计文档

**日期**: 2026-06-03
**状态**: 已确认

---

## 1. 概述

对 `feishu/push_events.py` 中全部 5 种飞书卡片模板进行格式整改，统一信息密度、引入机器类型配色、优化排版层次。

---

## 2. 卡片 1：⏰ 任务提醒（蓝色 → 机型色）

**Builder**: `_build_reminder_card()`（push_events.py:459）
**事件**: `task_impending_start`、`task_start`、`task_impending_end`、`task_end`
**目标**: 仅小组长私信

### 2.1 改动

| 改动 | 说明 |
|------|------|
| Header 颜色动态化 | 根据 `machines.type` 查 `color_settings.type_colors` 取对应色 → 设置 `header.template`；未配置则回退 `"blue"` |
| 新增字段：排班时长 | `end_min - start_min`，格式化为 "1h10m" / "52m"，无数据则不显示 |
| 新增字段：优先级 | SQL JOIN `tasks.priority`；无数据则不显示 |
| 新增字段：所属来源 | 已有 `package_name`，有则显示任务包名，无则不显示 |
| 文案修改 | "提前X分钟" → "剩余X分钟" |
| 合并发送 | 同一轮检测周期内的 `individual_events` 按 `(target_type, target_id)` 合并为一张卡片；header 标题改为 "⏰ 任务提醒（N项）"；每条之间虚线分隔 |

### 2.2 卡片结构（改造后）

```
┌─────────────────────────────────────┐
│ ⏰ 任务提醒（N项）      [机型色]    │
├─────────────────────────────────────┤
│ 机器      │ 任务                    │
│ 时间      │ 状态（剩余X分钟）       │
│─────────────────────────────────────│
│ ⏱ 时长  ⚡ 优先级  📦 来源  🏷 分组│
│──────────────── ─ ─ ─ ─ ─ ─ ─ ─ ─ ─│
│ （下一条…）                         │
└─────────────────────────────────────┘
```

### 2.3 数据依赖

- `machines.type` → `color_settings.type_colors` 查颜色
- `tasks.priority` → 需在 `_detect_feishu_events` SQL 中追加 JOIN
- `event.package_name` → 已有
- `event.start_min / end_min` → 已有，计算时长

## 3. 卡片 2：✅ 任务动态公告（绿色）

**Builder**: `_build_announcement_card()`（push_events.py:494）
**事件**: `task_confirm_start`、`task_confirm_end`、`package_complete`
**目标**: 群 + 可选私信

### 3.1 改动

| 改动 | 说明 |
|------|------|
| 信息顺序统一 | 从 "机器 任务 动作" → "分组 → 机器 → (任务包) → 任务 → 动作"，有则显示无则跳过 |
| 色块标注 | 每行加浅色背景 + 左边框：蓝底=确认开始、绿底=确认完成、橙底=包完成 |
| 包完成统一 | 不再用独立 `div.text`，与其他行结构一致，只显示任务包名+动作 |

### 3.2 三种格式变体

| 场景 | 拼装结果 |
|------|---------|
| 有任务包 | `全能组 BR1-01 药品任务包 常规采集 已确定开始` |
| 无任务包 | `全能组 BR2-01 药盒右移 已确定完成` |
| 包完成 | `药品任务包 📦 全部任务已完成` |

### 3.3 色块规则

| 事件 | 底色 | 左边框 |
|------|------|--------|
| `task_confirm_start` | #E3F2FD | #2196F3 |
| `task_confirm_end` | #E8F5E9 | #4CAF50 |
| `package_complete` | #FFF8E1 | #FF9800 |

## 4. 卡片 3：📋 排班变动汇总（橙色）

**Builder**: `_build_changes_card()`（push_events.py:533）
**事件**: `schedule_changes`
**目标**: 群 + 小组长（颜色反转）

### 4.1 改动

| 改动 | 说明 |
|------|------|
| 上下文分组去重 | 相同（分组 + 机器号 + 任务包）共享灰色 header，下面每条任务只列自己的时间变化 |
| 双时间线展示 | 开始时间和结束时间各自独立展示：old→new (±diff 方向标签) |
| 新增变化后时长 | `new_end - new_start` 格式化 XhXm，数据不足显示 "—" |
| 颜色反转 | 群发：延后=橙/提前=绿；私信小组长：延后=绿/提前=橙 |

### 4.2 卡片结构（改造后）

```
┌─────────────────────────────────────┐
│ 📋 排班变动汇总                     │
├─────────────────────────────────────┤
│ 全能组 BR2-05 药品任务包  ← 上下文  │
│   晃动管状药膏                       │
│   开始时间 11:18→11:40 (+22m ⏰延后) │
│   结束时间 14:20→14:40 (+20m ⏰延后) │
│   变化后排班时长 3h20m               │
│                                     │
│   挤压管状药膏一次                   │
│   开始时间 —（未变化）               │
│   结束时间 14:26→15:00 (+34m ⏰延后) │
│   变化后排班时长 —                   │
│                                     │
│ 全能组 BR1-01            ← 无任务包  │
│   药盒右移紧贴创可贴                 │
│   开始时间 09:33→09:20 (-13m ⏫提前) │
│   结束时间 —（未变化）               │
│   变化后排班时长 57m                 │
└─────────────────────────────────────┘
```

### 4.3 数据依赖

- `old_end_min` → 已在 event dict 中（`_detect_feishu_events` 第 287 行），当前卡片未使用
- 每条按 start 变动方向决定色块（延后=橙/提前=绿/无变化=灰）

### 6.3 数据依赖

- 按 `package_id` GROUP BY 统计每包完成数/总数（复用 `models.py:list_task_packages()` 的查询模式，加 `schedules.date` 过滤）
- 需 `SUM(collect_count)` 统计采集条数
- 夜班跨两天查询逻辑保持不变
- 第三段不写异常原因（组长自行统计）

### 6.4 手动触发：立即进行班次总结

在飞书推送设置页面新增按钮"立刻进行班次总结"，点击后实时计算当前班次状态并推送。

**API**: `POST /api/feishu/push-config/report-now`

逻辑：
1. 判断当前班次（白班/夜班）
2. 调用 `_build_report_card()` 生成卡片
3. 向所有配置的群聊发送（不发私信，总结报告面向全群）

返回：
```json
{
  "shift": "白班",
  "results": [
    {"chat_id": "oc_xxx", "success": true},
    {"chat_id": "oc_yyy", "success": false, "error": "chat not found"}
  ]
}
```

**前端**：在推送设置 box 中，测试发送按钮旁新增"📊 班次总结"按钮。

## 7. 发送通道

两个发送通道不变：

| 函数 | API | 用途 |
|------|-----|------|
| `send_im_message()` | `POST /im/v1/messages?receive_id_type=chat_id` | 群聊 |
| `send_im_message_to_user()` | `POST /im/v1/messages?receive_id_type=open_id` | 小组长私信 |

所有卡片通过 `msg_type="interactive"` 发送。
