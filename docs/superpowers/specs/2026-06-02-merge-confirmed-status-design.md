# 统一 "已确认" → "已完成" 状态

## 概述

数据库中任务状态同时存在 "已完成" 和 "已确认" 两个等价值，但后端多处逻辑只认 "已完成"，导致任务包统计、回收保护、全完成包查询等功能对 "已确认" 任务失效。

**方案**：消除 "已确认" 写入源 + 数据库一次性迁移 + 前端保留守卫。

---

## 根因分析

`routes/schedule_ops.py:612`（确认过时任务快捷操作）是唯一写入 `"已确认"` 的路径。其他所有完成路径（`finish_task`、`batch_tasks`、任务拆分完成）都写 `"已完成"`。

后端 **8 处** 判断中只有部分同时检查 "已确认"：

| 文件:行 | 判断 | "已确认" 是否被正确处理 |
|---|---|---|
| `models.py:314` | 全完成包查询 | ❌ 不认 |
| `models.py:326` | 包已完成计数 | ❌ 不认 |
| `models.py:329` | 包已分配计数 `NOT IN` | ❌ 不认 |
| `models.py:382,386,391,395` | 包删除级联保护 | ❌ 不认 |
| `models.py:506` | 回收时不覆盖已完成 | ❌ 不认 |
| `models.py:542` | 切割段回收保护 | ❌ 不认 |
| `routes/schedule_ops.py:63` | 重新分配不覆盖已完成 | ❌ 不认 |
| `routes/tasks.py:280` | 回收入池对已完成的特殊处理 | ❌ 不认 |
| `task-table.js:89` | 任务库过滤隐藏 | ✅ 正确 |
| `task-status.js:81,91` | 动态状态不覆盖 | ✅ 正确 |

## 改动

### 第一层：消除写入源

**`routes/schedule_ops.py:612`**

```python
# Before
conn.execute("UPDATE tasks SET status=? WHERE id=?", ("已确认", int(tid)))

# After
conn.execute("UPDATE tasks SET status=? WHERE id=?", ("已完成", int(tid)))
```

**`routes/schedule_ops.py:616`**

```python
# Before
return jsonify({"msg": f"已确认 {confirmed} 个过时任务"})

# After
return jsonify({"msg": f"已标记完成 {confirmed} 个过时任务"})
```

### 第二层：数据库迁移

在 `db.py` 的 `init_db()` 末尾（表结构迁移之后）增加一段数据迁移，利用 SQLite 的 try/except 模式保证幂等：

```python
# 数据迁移：统一历史遗留的 "已确认" 状态
cur.execute("UPDATE tasks SET status='已完成' WHERE status='已确认'")
conn.commit()
```

该 UPDATE 天然幂等——首次执行后不再有 `'已确认'` 行，重复执行无影响。无需版本号判断。

### 第三层：前端文案更新 + 守卫保留

**`timeline-ops.js:188,258`** — 更新 UI 文案，把 "已确认" 改为 "已完成"：

```
# Before
将已过期但未完成的任务标记为"已确认"

# After
将已过期但未完成的任务标记为"已完成"
```

**`task-table.js:89`** — 保留不变（防御性守卫）：

```js
if (t.status === '已完成' || t.status === '已确认') continue;
```

**`task-status.js:81,91`** — 保留不变（防御性守卫）：

```js
if (dyn && task.status !== '已完成' && task.status !== '已确认') { ... }
if (dyn && orig !== '已完成' && orig !== '已确认') { ... }
```

### 不需要改的地方

后端 8 处只认 `'已完成'` 的判断**不做任何修改**——迁移后数据库干净了，现有逻辑自动正确。这些位置是：

- `models.py:314,326,329` — 任务包统计
- `models.py:382,386,391,395` — 包删除级联
- `models.py:506` — 回收保护
- `models.py:542` — 切割段回收
- `routes/schedule_ops.py:63` — 重分配保护
- `routes/tasks.py:280` — 回收入池特殊处理

---

## 对任务包的影响

迁移后，所有已完成任务在数据库中统一为 `status='已完成'`。任务包的以下功能**无需任何代码修改**即可自动修复：

- 包卡片 "已完成 X/总数" 统计（A 问题）
- 展开视图中已完成任务的状态显示（B 问题）
- 排班侧完成任务后包卡片进度更新（C 问题）
- 全部完成的包出现在历史记录面板
- 删除任务包时已完成子任务被正确保护
- 回收操作不会覆盖已完成任务的状态
