# 切割任务命名修正 & 删除段条数回收

**日期**: 2026-05-27
**状态**: 已确认

## 问题一：再切割命名叠加

当前 `schedule_cut.py` 每次切割创建新 `split_group`，且用 `task["name"]` 直接拼后缀，导致：

```
巡检（第二段）→ 再切割 → 巡检（第二段）（第一段）+ 巡检（第二段）（第二段）
```

**修正**：

1. 去旧后缀：`base_name = re.sub(r'（第[一二三四五六七八九十]+段）$', '', task["name"])`
2. 复用 group：原任务有 `split_group` 则沿用，否则新建 UUID
3. 整体重编号：删除原任务、插入两段后，对该 group 所有段按时间顺序重新分配 `split_order` 和名称

**效果**：

```
切割前:  [巡检（第一段）order=1, 巡检（第二段）order=2]
切割第二段后: [巡检（第一段）order=1, 巡检（第二段）order=2, 巡检（第三段）order=3]
```

## 问题二：删除段时条数丢失

当前删除某段时，该段的 `expected_count` 随 task 记录一起消失，总条数减少。

**修正**（后端 `delete_schedule` + `del_task` 两个 API 统一处理）：

`del_task`（`routes/tasks.py`）当前没有 split_group 处理逻辑，需要补齐。

```
删除某段
 ├─ 有下一段 → 条数加到下一段（不限状态）
 ├─ 无下一段、有前一段
 │    ├─ 前一段未完成 → 条数加到前一段
 │    └─ 前一段已完成 → 不加，直接删
 └─ 没有其他段（仅剩此一段）→ 直接删除
```

**效果**：

```
[第一段:30, 第二段:30, 第三段:40] 总100
              ↓ 删第二段
[第一段:30, 第三段:70] 总100 ✓

[第一段:30(已完成), 第二段:70]
              ↓ 删第二段
[第一段:30] 总30（已完成的任务，条数不加回）
```

## 问题三：删到只剩一段时清理命名

当 group 内只剩一个任务时，去掉 `（第X段）` 后缀：

```
[巡检（第一段）, 巡检（第二段）, 巡检（第三段）]
  ↓ 删第二段、第三段
[巡检]
```

**修正**：在两个 API 的 split_group 重编号逻辑中，重编号完成后检查 group 内数量，若仅剩 1 条则 `UPDATE tasks SET name = base_name`（去掉后缀）。

## 操作后 Toast 提示

后端在 response 中返回详情字段，前端直接用 `d.msg` 展示：

| 操作 | 后端 msg 示例 |
|------|-------------|
| 切割 | `切割完成：已创建「巡检（第二段）」「巡检（第三段）」` |
| 删除（回收给X段） | `已删除，30 条数据已回收至「巡检（第三段）」` |
| 删除（前一段已完成，不加回） | `已删除` |
| 删除（仅剩一段，去后缀） | `已删除` |

## 涉及文件

| 文件 | 改动 |
|------|------|
| `routes/schedule_cut.py` | 去旧后缀、复用 group、整体重编号、完善 msg |
| `routes/schedule_ops.py` `delete_schedule` | 条数回收、仅剩一段清后缀、完善 msg |
| `routes/tasks.py` `del_task` | 新增 split_group 重编号、条数回收、仅剩一段清后缀、完善 msg |

## 不影响

- 前端：逻辑上无需改动（后端 msg 会自动显示在 toast 中）
- 撤销切割（`undo_cut`）
- 完成分段任务（`complete_split_task`）
- 切割本身的时间分配逻辑
