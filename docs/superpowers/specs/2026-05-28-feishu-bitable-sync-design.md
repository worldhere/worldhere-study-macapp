# 飞书多维表格双向同步 — 设计规格

## 概述

将排班系统的排班记录与飞书多维表格打通，一个机器对应一张飞书表。操作员在飞书端查看机器排班、填写实际时间、标记异常，系统端做复杂排班管理和统计。双向自动同步。

**核心理念**：系统是唯一数据源，飞书是操作窗口。飞书删行不等于系统删数据。

## 数据模型变更

### schedules 表新增字段

```sql
ALTER TABLE schedules ADD COLUMN actual_start_min INTEGER;
ALTER TABLE schedules ADD COLUMN actual_end_min INTEGER;
```

- `actual_start_min`：实际开始绝对分钟数，可空
- `actual_end_min`：实际结束绝对分钟数，可空

预估时间（`start_min` / `end_min`）与实际时间（新增字段）独立存储。

### 同步映射表

```sql
CREATE TABLE IF NOT EXISTS feishu_sync_mapping (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    machine_id INTEGER NOT NULL UNIQUE,
    machine_name TEXT NOT NULL,
    app_token TEXT NOT NULL,
    table_id TEXT NOT NULL,
    last_pull_at TEXT,
    last_push_at TEXT
);
```

记录每台机器对应的飞书表格信息及最近同步时间。

## 飞书表格结构

### 每台机器一张表，字段如下：

| 列名 | 飞书类型 | 必填 | 来源 | 说明 |
|---|---|---|---|---|
| 任务名 | 文本 | — | 系统 | 排班对应的任务名称 |
| 任务类型 | 文本 | — | 系统 | BR1/BR2/Mini |
| 优先级 | 文本 | — | 系统 | P0/P1/P2 |
| 难度 | 文本 | — | 系统 | 无/简单/普通/困难 |
| 预估开始 | DateTime | — | 系统 | 含日期+时间，跨天自动换算 |
| 预估结束 | DateTime | — | 系统 | date + timedelta(minutes=end_min) |
| 预估时长 | 文本 | — | 系统（只读） | 系统根据预估秒数自动计算，如 "2h"、"1h30m" |
| 实际开始 | DateTime | 否 | 用户 | 操作员填，用于计算耗时 |
| 实际结束 | DateTime | 填了就变已完成 | 用户 | 操作员填，填完状态自动变"已完成" |
| 状态 | 文本 | — | 飞书自动+双向 | "执行中" / "已完成" |
| 排班备注 | 文本 | 否 | **双向** | 系统分配时写入，飞书端也可编辑回传 |
| 异常标记 | 单选 | 否 | 用户 | "正常" / "机器故障" / "缺少物料" / "无法执行" |
| 异常备注 | 文本 | 否 | 用户 | 故障时描述原因 |
| _记录ID | 文本 | — | 系统（隐藏） | 格式 `schedule_{id}`，用于匹配飞书行与系统记录 |

默认视图按「预估开始」升序排列。飞书的筛选和排序为视图层操作，不影响 API 数据的读写。

### 耗时计算

当状态变为"已完成"时，计算实际耗时：

| 实际开始 | 计算方式 |
|---|---|
| 已填写 | `实际结束 - 实际开始` |
| 未填写 | `实际结束 - 预估开始`（保底） |

### 状态联动

- 飞书端填写实际结束时间 → 状态自动变"已完成" → 推送到系统
- 系统端标记完成 → 飞书行状态更新为"已完成"

## 同步策略

### 系统 → 飞书（防抖批量推送）

每次排班变更（分配、移动、完成、编辑、切割、删除）进入脏标记队列：

- **防抖窗口**：10 秒内无新变更即推送
- **上限**：自首次脏标记起 30 秒强制推送
- **防抖/上限由前端 JavaScript 定时器驱动**，后端提供 `/api/feishu/push` 接口

#### 推送范围（基于真实日期）

基于系统当前真实日期，推送范围：

- **往后**：今天 + 7 天
- **往前**：今天 - 3 天（包括今天）

即每台机器推送 `[今天-3天, 今天+7天]` 区间内的所有排班记录。

#### 行数优先级（200 行上限）

当区间内排班超过 200 行时，按优先级裁剪：

| 优先级 | 日期范围 | 说明 |
|---|---|---|
| 1（最高） | 今天 | 全部保留 |
| 2 | 昨天 + 明天（今天 ± 1 天） | 全部保留 |
| 3 | 剩余区间按离今天远近 | 越近优先，最远的新/旧行先被丢弃 |

裁剪时丢数据只丢飞书端，系统数据不动。

#### 推送逻辑（按 `_记录ID` 匹配合并）

| 匹配情况 | 动作 |
|---|---|
| 系统有、飞书有 | 更新系统侧字段（任务名、任务类型、优先级、难度、预估开始/结束、预估时长、状态），**不动**用户侧字段（实际开始/结束、异常标记、异常备注）。排班备注为双向字段：推送时覆写，但拉取时若飞书端有更新也会回传覆盖 |
| 系统有、飞书无 | 新增行，用户侧字段留空 |
| 系统无、飞书有 | 删除飞书行 |
| 系统无排班 | 清空该机器飞书表所有行 |

### 飞书 → 系统（定时轮询）

由前端 `setInterval` 每 30 秒触发：

1. 调用 `/api/feishu/pull`，遍历所有已映射机器，读取飞书表记录
2. 比对字段：实际开始、实际结束、状态、排班备注、异常标记
3. 有差异更新本地 `schedules` 表
4. 返回变更摘要给前端，前端据此刷新相关机器的时间线

### 冲突处理

推送前先拉一次目标表格，若发现飞书侧有未同步的编辑，先应用飞书侧变更到本地，再执行推送。简化为最后写入者胜出。

## 行数上限与清理

| 规则 | 条件 | 行为 |
|---|---|---|
| 日期窗口 | 排班日期超出 `[今天-3天, 今天+7天]` | 推送时自动排除（下一次窗口缩进就回到范围内，重新出现） |
| 单表上限 | 窗口内排班超过 200 行 | 按优先级裁剪：今天全部 > 今天±1天 > 近 > 远 |
| 系统删除任务 | 推送时发现 `_记录ID` 在系统已不存在 | 从飞书表移除对应行 |

**飞书删行 ≠ 系统删数据**，系统数据库永远保留完整记录。

## 异常标记与系统联动

### 三种异常类型及其系统行为

| 异常标记 | 系统行为 | 飞书表现 |
|---|---|---|
| 机器故障 | 触发维修停用模块，记录停机时段（category="机器故障"） | 该行保留，异常标记显示"机器故障" |
| 缺少物料 | 触发维修停用模块，记录停机时段（category="缺少物料"） | 该行保留，异常标记显示"缺少物料" |
| 无法执行 | 回收任务到待分配池 + 压实后续任务 | 该行从飞书表**消失**（任务已不属于这台机器） |

### "机器故障"/"缺少物料" 流程

```
飞书标记异常
  → 30s 轮询拉回系统
  → 系统调用维修模块：开始停机，记录开始时间、原因类目
  → 飞书对应行保留，操作员可见机器状态

飞书恢复为"正常"（操作员修复后改回）
  → 30s 轮询拉回系统
  → 结束维修记录，计算停机时长
  → 全局开关 ON：自动延后受影响任务的结束时间（后期做）
  → 全局开关 OFF：仅记录，手动调整
```

### "无法执行" 流程

```
飞书标记"无法执行"
  → 30s 轮询拉回系统
  → 系统执行回收（任务回到待分配池）
  → 系统执行压实（后续任务前移，start_min/end_min 自动更新）
  → 下一轮推送（走正常防抖/上限策略，不跳过）
  → 飞书表：被回收行消失，剩余行预估时间自动前移
```

操作员视角：标记"无法执行"后，刷新飞书表，被回收的任务行不见了，后面的任务时间都往前挪了。

### 异常通知推送

```
异常标记变更（正常→异常）
  → 30s 轮询拉回系统 → 识别为首次异常
  → 触发通知（推送细节另立规格）
  → 已是异常或重复标记 → 跳过
```

异常标记恢复（异常→正常）也触发通知，告知相关人机器已可用。

---

本规格只保留异常标记/异常备注字段及上述联动逻辑，IM 通知推送细节另立规格。

## 同步总开关

设置页面提供一个开关，控制所有飞书数据传输的启停。

### 行为

| 开关状态 | 推送 | 轮询 | 初始化 |
|---|---|---|---|
| 关闭（默认） | 跳过 | 跳过 | 跳过 |
| 开启 | 正常运行 | 正常运行 | 允许 |

- 配置键：`feishu_sync_enabled`，存入 `config` 表，默认 `"0"`
- 前端所有飞书同步逻辑（`markDirty`、setInterval 轮询、防抖触发）在执行前检查此标志
- 后端 `/api/feishu/push` 和 `/api/feishu/pull` 同样检查此标志，双重保险

## 全时段初始化（增量校验）

初始化不是一次性批量操作，而是每次启动时的增量校验。触发时机：

- 系统启动
- 飞书开关从关→开
- 机器新增/删除/重命名

### 校验流程

```
遍历所有机器
  ├─ 检查 feishu_sync_mapping 是否有记录
  │     ├─ 无 → 飞书侧查是否有同名表
  │     │        ├─ 有 → 直接建立映射，写入 mapping 表
  │     │        └─ 无 → 创建表 + 建映射
  │     └─ 有 → 验证 mapping.table_id 在飞书是否仍存在
  │              └─ 已删除 → 重建 + 更新 mapping
  ├─ 检查飞书表字段是否与系统定义一致
  │     ├─ 缺字段 → 补加
  │     ├─ 多字段 → 忽略（不自动删，防误删用户数据）
  │     └─ 异常标记选项不一致 → 用系统配置覆写
  └─ 检查废弃映射（机器已删但 mapping 还在）
        └─ 标记 stale，不自动删飞书表（手动决定）
```

### 原则

- 不急不躁，慢慢对齐
- 只补不缺，不自动删表
- 每次操作都有日志，状态接口可查

## 异常标记选项管理

异常标记的单选选项由系统**统一配置**，不直接在飞书改。

### 配置存储

`config` 表中键 `feishu_exception_options`，JSON 字符串格式：

```json
["正常", "机器故障", "缺少物料", "无法执行"]
```

### 管理入口

设置页面 → 异常标记选项 → 增/删/改选项

### 生效时机

选项变更后 → 下次全时段初始化触发 → 更新所有机器飞书表字段的选项列表

### 推送保护

若飞书端异常标记值不在当前选项列表中（旧值已被删除），保留原值不变，不强制清零。在 status 接口标出作为提醒。

## 实际时间校验

拉取实际开始/实际结束时做校验：

```
if actual_start and actual_end and actual_start > actual_end:
    → 不更新系统
    → 记入 status 接口的 validation_errors 列表
    → 飞书端行本身不动（下次拉取继续尝试）
```

错误信息格式：

```json
{ "machine": "BR1-01", "record_id": "schedule_42", "error": "实际开始晚于实际结束" }
```

机器生命周期联动时自动触发增量校验。

| 系统操作 | 飞书侧 |
|---|---|
| 新增机器 | 调用 API 创建表格（在同一 Base 下），写入 `feishu_sync_mapping` |
| 删除机器 | 仅删除映射，保留飞书表格（手动决定是否删） |
| 重命名机器 | 更新飞书表格名称 |

## 异常标记联动维修模块

`/api/feishu/pull` 检测到异常标记变化时直接调用维修模块：

```
异常标记: 正常 → 机器故障
  → models.start_repair(machine_id, category="机器故障")

异常标记: 正常 → 缺少物料
  → models.start_repair(machine_id, category="缺少物料")

异常标记: 机器故障/缺少物料 → 正常
  → models.end_repair(machine_id)
  → 记录停机时长，全局开关控制是否自动延后排班（后期做）
```

## Token 管理

后端直调飞书 API 时需自行管理 `tenant_access_token`：

- App ID + App Secret → `auth/v3/tenant_access_token/internal`
- 有效期 ~2 小时，内存缓存 + 过期前 5 分钟提前刷新
- API 返回 401 时立即刷新重试一次

```python
# feishu_token.py
token_cache = {"token": None, "expires_at": 0}

def get_token():
    if time.time() > token_cache["expires_at"] - 300:
        resp = requests.post("https://open.feishu.cn/open-apis/auth/v3/...", ...)
        token_cache["token"] = resp["tenant_access_token"]
        token_cache["expires_at"] = time.time() + resp["expire"]
    return token_cache["token"]
```

## 任务切割兼容

切割后每段有独立 schedule ID，`_记录ID = schedule_{id}` 天然区分，无需额外处理。

切割前后飞书表现：

```
切割前：1 行（预估 21:00-21:30）
切割后：2 行（预估 21:00-21:15 + 21:15-21:30）
```

推送时按 `_记录ID` 匹配，旧行消失，两行新行出现。

## API 重试策略

| 错误类型 | 策略 |
|---|---|
| 网络超时 / 连接错误 | 3 次重试，间隔 1s / 2s / 4s |
| 429 频率限制 | 等待 Retry-After 头，最多 5 次 |
| 401 token 过期 | 刷新 token 后重试 1 次 |
| 4xx（字段/参数错误） | 不重试，记错误日志到 status.retry_log |
| 5xx | 3 次重试，仍失败则跳过本次，下个周期再试 |

## 新增路由

| 路由 | 方法 | 用途 |
|---|---|---|
| `/api/feishu/init` | POST | 增量校验：对齐所有机器与飞书表的映射、字段、选项 |

## 边界与限制

- 飞书 API 频率：单应用 100 次/秒，30 秒轮询远低于阈值
- 单表最大行数 200，溢出时暂停写入并优先清理最旧的已完成任务腾位
- 离线时推送失败，下次防抖触发时自动重试
- 首次初始化全量创建 100 台机器表格，预计耗时若干分钟，提供进度提示
