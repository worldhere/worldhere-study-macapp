# Golden 排班系统 — Web 化可行性研究报告

> 调研时间：2026-06-12
> 调研目标：让排班系统从单机 `127.0.0.1:5000` 变为可被其他电脑通过浏览器访问
> 约束条件：不能内网穿透、不能云服务器、不能外部中转服务

---

## 一、项目现状

| 维度 | 现状 |
|------|------|
| 后端 | Flask + SQLite (WAL 模式) |
| 前端 | 原生 JS，全部相对路径 API 调用 |
| 当前绑定 | `host='127.0.0.1'`，仅本机可访问 |
| 多用户并发 | WAL 模式下 2-10 人同时读写无问题 |
| 多标签稳定性 | 各做各的无问题，同数据同时编辑则后者胜出 |
| 前端跨主机风险 | 零——无硬编码 URL，无跨域资源加载 |

---

## 二、网络环境约束

```
✅ 一台 Windows 工作站（可长期运行）
✅ 所有机器能上网
✅ 所有机器装有飞连客户端（corplink 虚拟网卡已存在）
✅ 工作站飞连预定 IP：10.77.151.32
❌ 不能内网穿透（公司安全策略禁止）
❌ 不能云服务器（审核不通过）
❌ 不能外部中转（Cloudflare Tunnel 等，等同于云服务器，审不过）
❌ 局域网隔离（Windows 工作站在不同 VLAN，互 ping 不通）
❌ 同事不是飞连账号拥有者，不能开启飞连 VPN
```

---

## 三、探索过的方案

### 方案 A：局域网直连

**原理：** 所有机器在同一网段，通过 IP 直接访问 Flask。

**路径：** `同事浏览器 → http://工作站的局域网IP:5000`

**结论：** ❌ 不可行

**原因：** Windows 工作站被 IT 划入不同 VLAN，与 Linux 工作站网络隔离，互 ping 不通。

---

### 方案 B：内网穿透（端口映射 / frp / ngrok 等）

**原理：** 在公网开一个端口或中转节点，外部流量经此进入工作站。

**结论：** ❌ 禁止

**原因：** 公司安全策略明确禁止内网穿透。

---

### 方案 C：云服务器部署

**原理：** 把 Flask 应用部署到阿里云/腾讯云/AWS 等云服务器上。

**结论：** ❌ 审核不通过

**原因：** 云服务器采购/使用需要审批，大概率不批。

---

### 方案 D：外部中转（Cloudflare Tunnel 等）

**原理：** 工作站出站连接到 Cloudflare 边缘节点，外部用户通过 Cloudflare 域名访问。

**结论：** ❌ 等同于云服务器

**原因：** 本质上使用外部云服务做中转，审核标准与云服务器相同，过不了审。

---

### 方案 E：飞连 VPN 模式（直连）

**原理：** 所有人打开飞连 VPN，通过飞连分配的虚拟 IP 直接访问 Flask。

**路径：** `同事开飞连VPN → 获得虚拟IP → 浏览器访问 http://10.77.151.32:5099`

**结论：** ❌ 不可行

**原因：** 同事不是飞连账号拥有者，不能开启飞连 VPN 开关。

---

### 方案 F：飞连应用网关模式 ✅ 推荐

**原理：** 飞连网关作为 L7 反向代理，同事通过浏览器访问飞连托管域名，网关验身份后转发到工作站。

**路径：**
```
同事浏览器 → https://xxx.feilian.cn → 飞连应用网关 → http://10.77.151.32:5099
```

**优势：**
- 同事零负担：不需要飞连客户端、不需要 VPN、浏览器打开链接即可
- 工作站零暴露：内网 IP 完全隐藏在网关之后
- 公司已有基础设施：飞连是已采购平台，应用网关是标准功能
- 合规：走公司正规 IT 流程，不涉及任何外部服务
- 代码改动极小：仅改 `app.py` 一行（host + port）

**需要满足的前提：**
- 飞连网关节点能路由到 `10.77.151.32`（IT 配的时候能验证）
- 工作站飞连客户端保持在线

**IT 需要做的操作：**
1. 在飞连管理后台「应用网关 → Web 应用」新建应用
2. 后端地址填写 `http://10.77.151.32:5099`
3. 选择飞连托管域名（填一个前缀即可）
4. 配置授权范围（指定人员）
5. 选择免认证或 SSO 认证

## 六、实测验证：多用户并发压力测试

> 测试时间：2026-06-12
> 测试脚本：`stress_test.py`（10 线程 × 8 种用户行为模式）
> 测试脚本：`network_test.py`（6 种网络异常场景模拟）

### 6.1 多用户并发测试

| 指标 | 数值 |
|------|------|
| 并发线程数 | **10**（8 种不同行为模式） |
| 总请求数 | **863** |
| 成功率 | **100%** (863/863) |
| 吞吐量 | **200.8 req/s** |
| 写操作成功率 | **100%** (273/273) |
| 读操作成功率 | **100%** (590/590) |
| 慢查询（>5s） | **0 个** |
| 数据库完整性 | **OK**（测试后 PRAGMA integrity_check 通过） |
| WAL 状态 | 干净（0 bytes，全部 checkpoint） |

**10 个并发用户覆盖的操作类型：**

| 用户角色 | 操作轮数 | 行为 |
|---------|---------|------|
| 读密集型 ×2 | 各 30 轮 | 连续读取 8 类 API（machines/tasks/settings/schedules/history/packages/groups/deletion_log） |
| 排班操作 | 20 轮 | assign → move → complete → uncomplete 完整生命周期 |
| 设置变更 | 15 轮 | 枚举 add → update → reorder → delete 全流程（4 种枚举类型） |
| 导出用户 | 5 轮 | Excel 导出（executing + completed）+ history 查询 |
| 机器管理 | 15 轮 | 状态变更（空闲 ↔ 工作 ↔ 维修停用），联级异常标记 |
| 批量操作 | 10 轮 | recycle_uncompleted + confirm_overdue |
| 任务包操作 | 8 轮 | 创建 → 加任务 → 改名（PUT）→ 级联删除 |
| 混合 CRUD ×2 | 40 轮 | 70% 随机读 + 30% 随机写 |

**关键发现：**
- SQLite WAL 模式完美扛住 10 并发：`busy_timeout=5000` 从未触发，零超时
- 设置并发变更无冲突：不同 key 互不干扰，同一 key 后者胜出（`ON CONFLICT DO UPDATE`）
- 排班 LWW 策略正确：assign/move/complete/uncomplete 全链路无报错
- Excel 导出不受并发影响：5 轮导出全部成功，平均 65ms
- 机器状态联级操作安全：维修标记 → 异常通知 → 状态恢复链路正常
- 863 次混合读写后数据库完整无损

### 6.2 网络异常测试

| 场景 | 模拟操作 | 结果 |
|------|---------|------|
| 客户端断连 | 发送 assign 请求后立刻关闭 socket | ✅ 服务端完整执行，数据已写入 |
| 重复点击 | 同一任务分配两次 | ✅ 幂等（同 task+machine 自动去重） |
| 慢查询期间并发读写 | Excel 导出期间其他用户读 API | ✅ WAL 读不阻塞，5 次查询延迟 4-22ms |
| 两人同时编辑同一排班 | A 改名 + B 改备注，同时发送 | ✅ LWW 后者胜出，无报错 |
| 不同 key 并发写 | 两个用户同时改不同设置项 | ✅ 互不干扰，各自写入成功 |
| 同一 key 并发写 | 两个用户同时改同一个设置项 | ✅ 后者胜出，不丢数据 |

**网络中断时的三种情况：**
1. 请求未到达服务器 → 什么都没发生，重试即可
2. 服务器收到并处理完，响应无法返回 → 数据已写入，重试幂等
3. 用户重复提交 → 核心写操作有去重逻辑（同 task+machine 的 assign 自动去重）

### 6.3 交互 Bug 防护机制

系统具备六层防护，单点异常不会扩散：

| 层级 | 机制 | 作用 |
|------|------|------|
| ① | try/catch（100+ 处） | 局部异常不崩全局 |
| ② | Undo/Redo（50 步栈） | 手动撤销 5 种操作（完成/取消/回收/删除/切割） |
| ③ | `_silentRefresh` | 每次数据变更后从服务端全量重建 UI，自动修复不一致 |
| ④ | WAL + integrity_check | 崩溃不掉数据，启动时自动检查 |
| ⑤ | deletion_log | 删除操作先备份再删除，可恢复 |
| ⑥ | 自动存档 | 后台定时全量 DB 快照，出事可读档恢复 |

## 七、已知限制与风险

| 风险点 | 影响 | 缓解措施 |
|--------|------|---------|
| 工作站 IP 变化 | 网关转发失效 | 确认 `10.77.151.32` 是否为固定分配 |
| 无应用认证 | 拿到链接的人都能访问 | 飞连网关做入口认证；或后续在 Flask 中加登录 |
| Werkzeug 开发服务器 | 10 人以上可能不稳定 | 换成 waitress（一行代码） |
| 配置/样式不同步 | 一人改了颜色，其他人不刷新看不到 | 影响小，手动刷新即可 |
| 飞连策略未知 | 网关节点可能不通工作站 IP | IT 配的时候会验证 |
