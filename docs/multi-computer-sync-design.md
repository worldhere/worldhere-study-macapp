# 排班系统多机同步方案设计

> 从"为什么坏了"到"怎么做才对"的完整思考链路。

---

## 一、起点：一次故障揭示了什么

2026年6月5日上午，飞书发来一条通知：

> ✅ 异常恢复 — BR2-33 — 机器故障 — 开始 06-04 19:23 — 结束 10:42 — 异常总耗时：（空）

随后用户发现：任务库、排班面板、历史记录全部显示"加载失败"。排查后发现是 `filter-system.js` 重构时遗漏了两个全局变量（`_taskFilterState` 和 `_histFilterState`），跟飞书通知纯粹是巧合。

但在排查过程中，顺藤摸瓜发现了更值得关注的东西——

### 1.1 同一异常被"恢复"了两次

```
昨天 19:22 → schedule 122 (BR2-33) 标记"机器故障"
           → start_repair → repair_log 创建
昨天 19:25 → 异常清除 → end_repair → repair_log 关闭（修了3分钟）
           ⚠️ 飞书上的 exception_mark 没有同步清掉
今天 10:42 → 飞书 sync 再次发现"机器故障"→"正常"
           → _schedule_is_active = False（排班已过期）
           → 跳过 repair_log 操作
           → 但 _detect_exception_end 仍然生成了通知
```

这引出了一个根本问题：**飞书 sync 在跨天场景下会产生"幽灵事件"。** 排班昨天就结束了，今天还在检测。

### 1.2 如果两台电脑同时开了同步呢？

顺着这个思路往下推——当前系统假设全世界只有一台电脑在操作。如果两台呢？

翻代码发现了三个致命问题：

**问题一：push 会删别人的数据**

```python
# schedule_sync.py line 388
to_delete = [item["record_id"] for item in feishu_map.values()] + orphan_record_ids
#                                                                    ^^^^^^^^^^^^^^^^
#                 没有本地映射的飞书记录 = "脏数据" → 删掉
#                 但在多机场景下，那是另一台电脑的排班
```

A 和 B 各自 push 到同一张飞书表。A push 时，B 的记录没有本地映射 → 被当作"孤儿"删掉。B 下次 push 时删 A 的。**无限循环互相删。**

**问题二：pull 只更新不创建**

```python
# schedule_sync.py line 589
schedule_id = record_to_schedule.get(record_id)
if not schedule_id:
    continue  # ← 别人推上来的新排班，直接跳过
```

**问题三：通知双发**

A 和 B 都有事件检测引擎，各自有本地 push_log 表。同一条任务"即将开始"，两边都检测到，两边各发一张卡片。

### 1.3 三种场景推演

继续往下推，两台电脑的数据状态有三种情况：

| 场景 | 数据同步 | 通知 | 结论 |
|------|:---:|:---:|------|
| 不同数据库 | 🔴 互相删 | 🔴 双发 | 雪崩 |
| 克隆数据库（复制 SQLite 文件） | 🟢 基本正常 | 🔴 双发 | 热备可用，通知需修 |
| 克隆 + 只有一台操作 | 🟢 完美 | 🟡 只有一台发 | 已经是半手动热备方案 |

**第一次收敛**：如果两边的 SQLite 是一样的（克隆），问题一就不存在（所有飞书记录都有映射），问题三通过只在 Admin 端发通知解决。

这引出了第一个可行方向：**Admin-Client 分层**。

## 二、方向一：Admin-Client 分层

管理版 = 全部权限，客户端 = 查看 + 局部修改。

```
管理版 (Admin)                   客户端版 (Client)
─────────────                    ──────────────
✅ 全部 CRUD                      ✂️ 只读 + 局部可写
✅ 事件检测 → 通知派发             ❌ 不跑事件检测
✅ 数据校验                       ⚡ 能改: 排班时间/实际开始结束/异常标记/备注
✅ 存档备份                       ⚡ 能看: 排班/机器/任务包进度
```

**权限矩阵**：

| 操作 | Admin | Client |
|------|:-----:|:------:|
| 创建/删除任务 | ✅ | ❌ |
| 分配/回收任务 | ✅ | ❌ |
| 修改配置/班次 | ✅ | ❌ |
| 修改排班时间 | ✅ | ⚡ |
| 填写实际开始/结束 | ✅ | ⚡ |
| 标记异常/正常 | ✅ | ⚡ |
| 修改备注 | ✅ | ⚡ |
| 查看全部 | ✅ | ✅ |

这个方向解决了通知双发（只有 Admin 发），但数据同步的问题还在——如果两边的数据库不一样，互相删的问题没解决。

## 三、"云 U 盘"思路

此时用户提出了一个完全不同的想法：

> "单独弄个多维表格出来当作中转数据库，把 SQLite 数据搞到飞书表格里，然后再下载过来。像免费云 U 盘那样。"

这跳出了"实时同步"的框框。飞书不再只是镜像，而是**数据中心**。

### 3.1 Hub 表全量同步

```
上传（A → Hub）:
  遍历本地 SQLite 每张表 → 全量写飞书记录（带 source_id 标记归属）

下载（Hub → B）:
  拉取飞书全量 → 跳过自己的 → 对比本地 → INSERT/UPDATE/DELETE
```

**ID 冲突**：每台电脑用不同 ID 段（A: 1-9999, B: 10000-19999）。

**优点**：简单可靠、离线友好、天然备份。
**缺点**：非实时、无自动冲突合并。

### 3.2 更进一步：修现有同步

如果不想全量搬运，只改三处代码也能让实时同步勉强跑多机：

| 修改 | 位置 | 内容 |
|------|------|------|
| push 不删孤儿 | `schedule_sync.py:388` | 去掉 `+ orphan_record_ids` |
| pull 创建新排班 | `schedule_sync.py:589` | 无映射的飞书记录 → 本地创建 |
| 通知单发 | `events/__init__.py` | 事件检测前检查是否为 Admin |

改完就能跑，但没有权威源、没有冲突解决、边界情况多。

### 3.3 再进一步：操作日志事件溯源

不同步数据状态，同步操作指令。每个操作（创建任务、分配、修改时间）记录成一条不可变日志，所有电脑按序重放。

这能实现完整审计和冲突精确标记，但实现复杂度太高，暂时放一边。

## 四、更激进的思路：去 SQLite

> "重写所有功能，数据永远围绕飞书，飞书连不上本地就没数据。所有不同电脑的使用程序只用飞书的唯一表。"

这个想法把复杂度降到了最低——没有本地数据库就没有同步问题。

### 4.1 程序拆分

```
飞书多维表格（唯一数据库）
  ├── 数据管理程序   → 下载/上传/重置/校验/结构迁移
  ├── 排班面板       → 查看 + 改时间 + 填实际 + 标异常
  ├── 任务库         → 增删改查任务
  ├── 机器管理       → 增删改机器
  └── 设置           → 改配置

每个程序 = 独立 HTML + 轻量代理（几十行 Python）
未来可合并为「管理后台」+「使用前端」
```

### 4.2 技术选型：轻量代理（推荐）

纯前端直连飞书 API 会暴露 token、遇到 CORS 问题。用一个几十行的 Python 代理透传请求：

```python
# proxy.py — 整个"服务器"
@app.route("/api/proxy/<path:subpath>", methods=["GET", "POST", "PUT", "DELETE"])
def proxy(subpath):
    return feishu_request(request.method, f"/{subpath}",
                          data=request.get_json(silent=True),
                          params=request.args.to_dict())
```

### 4.3 规模化模拟

但"去 SQLite"面临一个现实问题。假设三个月后：100 台机器，每台每小时 1 个任务，20 组 × 2 组长/班 × 2 班。

```
数据量:
  排班记录: 100台 × 24条/天 × 90天 = 216,000 条
  飞书免费版上限: 50,000 条/表
  超出率: 332%

解决: 活跃表只保留最近 20 天，已完成排班自动归档

页面加载:
  排班面板（3 天窗口）: 7,200 条 ÷ 500/页 = 15 次 API ✅

事件检测:
  活跃排班: ~300 条
  通知目标: 80 组长 + 20 群 = 100 个
  每 30 秒: 300条 × 2事件 × 100目标 = 60,000 次去重查询

  如果 push_log 在飞书上 → 60K 次 API = 2,000 QPS 💥
  如果 push_log 在本地 SQLite → 0 次 API ✅
```

**关键发现**：push_log 不能上飞书。它不是业务数据，不需要多机共享——它只是"我发过这条通知没"的记忆，留在本地就行。

## 五、最终收敛：现有系统即管理机 + 飞书原生客户端

此时用户做了一个关键的决策判断：

> "现有系统直接当管理机的话，不仅能直接进行管理，而且我不用费心思去整合新管理程序的东西，只需要开发客户端就好了。"

这跟系统里一段天然保护机制完美契合：

```python
# schedule_sync.py — 推送时保护飞书端用户编辑的字段
USER_FIELDS = ["实际开始", "实际结束", "异常标记", "异常备注"]

for uf in USER_FIELDS:
    val = ex_fields.get(uf)
    if val is not None and val != "":
        sys_fields[uf] = val  # 保留飞书上的值，不覆盖
```

这意味着管理机 push 时不会覆盖客户端在飞书上的修改。**管理机写权威数据（排班、分配），客户端写现场反馈（实际时间、异常），互不打架。**

### 5.1 最终架构

```
┌──────────────────────────────────────────────────────────────────┐
│                     飞书多维表格（唯一业务数据源）                   │
│                                                                  │
│  machines │ tasks │ schedules │ config │ repair_log │ ...         │
└──┬───────────────────────────────────────────────┬──────────────┘
   │                                               │
   │ push/pull (现有同步引擎)                        │ 直接读写 (轻量代理)
   │ 管理机写: 全部字段                               │ 客户端写: 时间/异常/实际
   │ 管理机读: 客户端修改                             │ 客户端读: 全部数据
   │                                               │
┌──▼──────────────────────────┐    ┌────────────────▼─────────────┐
│   管理机（现有系统）           │    │   客户端（全新开发）            │
│                              │    │                              │
│  完整功能，一行不改            │    │  排班面板（查看 + 有限编辑）     │
│  SQLite + 飞书同步引擎         │    │  没有本地数据库                 │
│  事件检测 → 通知派发（独跑）     │    │  通过轻量代理调飞书 API         │
│  存档系统（复用）              │    │                              │
└──────────────────────────────┘    └──────────────────────────────┘
```

### 5.2 管理机本地 SQLite 的新角色

```
现在（单机）:                       多机模式:
  SQLite = 唯一数据源                SQLite = 飞书的本地缓存
  存档 = 复制 SQLite 文件            存档 = 从飞书下载 → 写 SQLite → 存档文件
  回滚 = 替换 SQLite 文件            回滚 = 存档恢复 SQLite → 上传飞书
  push_log = 通知去重               push_log = 通知去重（本地，不参与同步）
```

存档系统可以直接复用：

```
存档:
  ① 从飞书拉取全量 → 写入本地 SQLite
  ② create_save() → .sqlite3 存档文件（现有逻辑）
  ③ rotate_autosaves() 轮转清理（现有逻辑）

恢复:
  ① restore_save() → 替换本地 SQLite（现有逻辑）
  ② 全量 push 到飞书
  ③ 客户端下次刷新 → 看到恢复后的数据
```

### 5.3 数据流

```
管理机操作（任务分配、配置修改）:
  写本地 SQLite → 30秒 push → 飞书 → 客户端刷新看到

客户端操作（改时间、填实际、标异常）:
  飞书 API 直接写 → 管理机 pull → 更新本地 SQLite → 事件检测 → 通知

通知:
  只有管理机一台发。push_log 在管理机本地，不占 API 配额。
```

### 5.4 为什么 push_log 必须留本地

```
push_log 在飞书上:
  60,000 次去重查询 / 30秒 = 2,000 QPS → 超过限额 20 倍

push_log 在管理机本地 SQLite:
  每次查 < 1ms，不计入 API 配额
  只有管理机发通知 → 不存在双发
  push_log 不是业务数据 → 不需要多机共享
```

### 5.5 要开发的内容

| 组件 | 说明 | 工作量 |
|------|------|--------|
| 轻量代理 | ~50 行 Python，透传飞书 API + token 管理 + 缓存 | 半天 |
| 数据管理程序 | 飞书 → SQLite（衔接存档），SQLite → 飞书（衔接恢复） | 半天 |
| 客户端排班面板 | 仿现有 UI，数据从代理拉，只暴露允许的操作 | 1-2 天 |
| 管理机改动 | **零** | 0 |

### 5.6 开发路线

```
第一步: 轻量代理
  → proxy.py（几十行）
  → 静态文件服务

第二步: 数据管理
  → 飞书 → SQLite（下载，给存档用）
  → SQLite → 飞书（上传，给恢复用）
  → 现有存档 UI 直接复用

第三步: 客户端排班面板
  → 纯 HTML+JS，通过代理读写飞书
  → 只暴露允许的操作权限

第四步: 修复现有实时同步（可选）
  → 方案三的三处改动
  → 只在需要"多台管理机互相看到对方数据"时做

第五步: 其他客户端页面（按需）
  → 历史记录、任务包进度、机器状态
```

## 附录 B：飞书 API 速查

### 核心概念

```
租户 (Tenant)     = 你的公司/组织
应用 (App)        = 在飞书开放平台注册的程序
APP_ID            = 应用身份证号 (cli_xxx)
APP_SECRET        = 应用密码（保密）
tenant_access_token = 用 APP_ID+APP_SECRET 换的临时通行证，2小时过期
APP_TOKEN         = 多维表格文档的 ID（URL 里那个 I7Izb...）

API 调用格式:
  GET /POST /PUT /DELETE
  https://open.feishu.cn/open-apis/bitable/v1/apps/{APP_TOKEN}/tables/{table_id}/records
  Header: Authorization: Bearer {tenant_access_token}
```

### 限额

| 维度 | 限制 | 当前使用 |
|------|------|---------|
| QPS | 100 次/秒/应用 | 峰值 ~5 |
| 每分钟 | 5,000 次 | 正常 < 100 |
| 每表行数 | 50,000（免费版） | 当前 ~300 |
| 分页 | 500 条/次 | — |
| 批量 | 500 条/次 | — |
| token 有效期 | 2 小时 | 自动续期 |
