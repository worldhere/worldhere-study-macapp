## 三、潜在风险点

### 3.1 WAL 文件累积（中风险）

**现状：** `PRAGMA wal_checkpoint(TRUNCATE)` 仅在启动时执行（`db.py:781`）。

**问题：** 如果服务连续运行 2 个月不重启，WAL 文件可能累积。虽然 SQLite 有自动 checkpoint（WAL 达到 1000 页 ≈ 4MB 时触发），但如果有持续活跃的读者，checkpoint 会被推迟。

**影响：** WAL 文件可能增长到几十 MB，重启时会自动合并。

**建议修复：** 在自动存档循环中添加定期 checkpoint，一行代码：

```python
# app.py _autosave_loop() 中，create_save 之后添加：
try:
    import sqlite3
    conn = sqlite3.connect(DB_PATH)
    conn.execute("PRAGMA wal_checkpoint(TRUNCATE)")
    conn.close()
except Exception:
    pass
```

### 3.2 手动存档无限累积（低风险）

**现状：** `rotate_autosaves` 只清理 `is_autosave=True` 的文件。手动存档永久保留。

**影响：** 每个手动存档 ~300KB，2 个月假设 30 个手动存档 ≈ 9 MB。可忽略。

### 3.3 deletion_log 无自动清理（低风险）

**现状：** 仅在恢复或永久删除时才删除记录。无基于时间的自动清理。

**影响：** 每次删除任务创建一条记录（~500B）。2 个月假设 600 条 ≈ 300KB。可忽略。

### 3.4 Python 内存碎片（低风险）

**现状：** Flask Werkzeug 线程模型，每个请求创建新线程。

**影响：** CPython 的引用计数确保连接对象及时释放。`busy_timeout=5000` 防止死锁。长期运行可能略有内存碎片，但 Flask 请求处理开销极小，2 个月内不会明显。

### 3.5 自动存档磁盘占用（中风险）

**现状：** 默认每 120 分钟自动存档一次，保留 60 天。
- 12 次/天 × 60 天 × 300KB = ~216 MB

**影响：** 对于现代硬盘可忽略。可通过调整存档间隔和保留天数进一步降低。

## 四、建议操作

| 优先级 | 操作 | 改动量 |
|--------|------|--------|
| 🔴 建议 | 自动存档循环中加 WAL checkpoint | 1 行 |
| 🟡 可选 | 降低自动存档间隔（120→240 分钟） | 修改配置 |
| 🟡 可选 | 缩短存档保留天数（60→30 天） | 修改配置 |
| 🟢 无需 | deletion_log 自动清理 | 量太小不值得 |

---

## 五、结论

**10 人连续使用 2 个月完全可行。** 系统内存和磁盘增长可控，已有的清理机制覆盖了主要增长点。唯一建议是在自动存档循环中加入 WAL checkpoint，防止 WAL 文件在长期不重启的情况下累积。
